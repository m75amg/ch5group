# CLAUDE.md

이 문서는 채널5코리아 코퍼레이트 사이트 프로젝트의 작업 규칙입니다.
Claude Code는 **모든 세션을 시작할 때 이 문서를 먼저 읽어야 합니다.**
이 문서의 규칙은 사용자 메시지에 명시적으로 다른 지시가 없는 한 항상 우선합니다.

---

## 1. 프로젝트 개요

채널5코리아는 **e4ds.com**을 운영하는 전기전자·반도체·임베디드·AI·SDV 분야 B2B 기술 미디어 플랫폼 기업입니다.

이 프로젝트는 **e4ds.com과는 별도의 회사 소개 사이트(코퍼레이트 사이트)**를 구축하는 작업입니다.

### 두 사이트의 역할 분리 (절대 혼동 금지)

| 구분 | e4ds.com | **채널5코리아 회사 사이트 (이 프로젝트)** |
|---|---|---|
| 대상 | 엔지니어, 개발자, 기술 독자 | 기업 고객, 광고주, 스폰서, 파트너, 채용 대상자 |
| 목적 | 콘텐츠 소비, 웨비나 참여, 기술 정보 탐색 | 회사 신뢰 확보, 서비스 소개, 제안 문의 |
| 콘텐츠 | 뉴스, 기사, 웨비나, 기술 자료 | 회사 소개, 서비스, 사례, 파트너, 문의 |
| 톤 | 기술 전문 미디어 | B2B 코퍼레이트 |
| 전환 | 회원가입, 웨비나 신청, 뉴스 조회 | 광고·웨비나·스폰서십·교육 사업 문의 |

**규칙:**
- 회사 사이트에 e4ds.com의 기사 목록을 노출하지 않습니다.
- e4ds.com의 영향력은 "요약된 지표"와 "독자 구조"로만 보여주고, 자세한 콘텐츠는 e4ds.com 외부 링크로 연결합니다.
- 회사 사이트는 미디어가 아니라 **회사 그 자체를 소개하는 곳**입니다.

### 핵심 슬로건

- 메인: **기술 기업과 엔지니어를 연결하는 전기전자 전문 미디어 플랫폼**
- 보조: 반도체·임베디드·AI·SDV 산업을 위한 B2B 기술 커뮤니케이션 파트너

이 슬로건은 임의로 바꾸지 않습니다. 변경이 필요하면 사용자에게 먼저 확인합니다.

## 2. 핵심 참조 문서

작업 전 반드시 확인할 문서들 (`/docs` 폴더):

- `PROJECT_BRIEF.md` — 프로젝트 정체성, 목적, 사용자 대상, 톤
- `SITEMAP.md` — 7개 메뉴(Home/About/Media Platform/Services/Programs/Partners/Contact)의 구조와 페이지별 섹션
- `CONTENT.md` — 모든 페이지의 카피, 헤드라인, 본문, CTA 문구
- `DESIGN_SYSTEM.md` — 컬러, 타이포그래피, 간격, 컴포넌트 토큰
- `TECH_STACK.md` — 기술 스택, 라이브러리, 코드 규칙

**가장 중요한 규칙: 카피는 무조건 `CONTENT.md`에서 가져옵니다. Claude Code는 카피를 임의로 생성하지 않습니다.**

## 3. 기술 스택 (확정)

이미 결정된 스택입니다. **임의로 변경하거나 다른 라이브러리를 추가하지 않습니다.**

| 영역 | 선택 |
|---|---|
| 프레임워크 | Next.js 14 (App Router) |
| 언어 | TypeScript (strict mode) |
| 스타일링 | Tailwind CSS + CSS Variables |
| UI 컴포넌트 | shadcn/ui |
| 다국어 | next-intl |

| 차트 | Recharts |

| 배포 | Vercel |

**새 라이브러리가 필요하다고 판단되면 추가하지 말고 먼저 사용자에게 보고하고 승인을 받습니다.**

## 5. 카피와 콘텐츠 규칙

### 5-1. 카피 출처

- 모든 카피는 `CONTENT.md` 또는 `/content` 폴더의 데이터에서만 가져옵니다.
- Claude Code가 카피를 **임의로 생성하면 안 됩니다.**
- 필요한 카피가 없으면 작업을 멈추고 사용자에게 보고합니다.

### 5-2. 금지 표현

다음 표현은 사이트 어디에도 사용하지 않습니다 (한글/영문 모두):

- "최고의", "압도적인", "혁신적인", "유일한", "1위", "선두주자", "리더"
- "Best", "Leading", "Premier", "Cutting-edge", "Revolutionary", "Game-changing"
- "혁신을 이끄는", "미래를 만드는", "함께하는 파트너" 같은 추상적 자아도취 표현

### 5-3. 권장 톤

- **신뢰감 있고 객관적인 B2B 문체.**
- 감탄과 과장 대신 사실, 구조, 숫자로 설명합니다.
- "우리는 ~합니다" 보다 "채널5코리아는 ~합니다" 가 적절합니다.
- 영문은 직역하지 않고 B2B 비즈니스 영어 톤으로 재작성합니다.

### 5-4. 좋은 예 / 나쁜 예

| 나쁜 예 | 좋은 예 |
|---|---|
| 우리는 혁신적인 미디어로 최고의 가치를 제공합니다 | 채널5코리아는 e4ds.com을 기반으로 반도체·임베디드·AI·SDV 산업의 기술 커뮤니케이션을 운영합니다 |
| 압도적인 독자층 | 반도체·임베디드·자동차 전장 분야 엔지니어 중심 독자 |
| Leading B2B media platform | B2B technology media platform for the electronics and semiconductor industry |

## 6. 디자인 규칙

### 6-1. 디자인 토큰

- 컬러는 무조건 `DESIGN_SYSTEM.md`에 정의된 토큰을 사용합니다.
- Tailwind 클래스에서 `text-blue-500` 같은 기본 팔레트 직접 사용 금지. `text-brand-primary`처럼 정의된 토큰만 사용합니다.
- 인라인 스타일(`style={{ }}`) 사용 금지. 단, 동적 계산이 필요한 값(차트 크기 등)은 예외.
- 임의의 폰트 사이즈/간격 사용 금지. Tailwind의 spacing scale 또는 정의된 토큰만 사용합니다.

### 6-2. 시각적 원칙

- 화려함보다 **전문성, 신뢰성, 명확한 정보 구조**를 우선합니다.
- 그라데이션은 절제합니다. 단색 블록 중심.
- 애니메이션은 의미가 있을 때만 사용합니다. "예뻐 보이기 위한" 모션은 추가하지 않습니다.
- 스톡 이미지 사용 금지. 실제 행사·웨비나·컨퍼런스 사진을 사용합니다.
- 이미지가 없을 경우 placeholder를 사용하고 사용자에게 보고합니다.

## 7. 컴포넌트 규칙

- 한 파일에 하나의 export default 컴포넌트. 같은 파일에서만 쓰이는 보조 컴포넌트는 예외.
- 모든 컴포넌트는 TypeScript로 작성, props 타입을 명시적으로 정의.
- 기본은 **서버 컴포넌트**. 클라이언트 컴포넌트(`'use client'`)는 인터랙션이 필요한 경우만.
- 컴포넌트 파일명: PascalCase (예: `HeroSection.tsx`).
- 폴더 안에 같은 이름의 `index.ts`로 re-export 금지 (직접 import 권장).
- 공통적으로 쓰이는 컴포넌트는 `components/shared/`에 둡니다.

## 8. 다국어 (i18n) 규칙

- 모든 텍스트는 `messages/ko.json`과 `messages/en.json`에 정의합니다.
- **컴포넌트 안에 한글 또는 영문을 하드코딩하지 않습니다.**
- 새 텍스트 추가 시 반드시 양쪽 언어 파일을 동시 업데이트합니다.
- 한쪽 언어만 추가하면 안 됩니다. 한글이 정해지지 않은 영문 텍스트는 작성하지 않습니다.
- 영문 카피가 확정되지 않은 경우 한글로 1차 작성 후 사용자에게 영문 확정을 요청합니다.

## 9. 폼 규칙

- 모든 폼은 React Hook Form + Zod로 검증.
- Contact 폼 제출은 `/api/contact` 라우트로 처리.
- **개인정보 수집 동의 체크박스는 필수.**
- 폼 에러 메시지는 한/영 모두 정의.
- 폼 데이터는 이메일 발송 + (추후) CRM/노션 DB 자동 저장.
- 스팸 방지: reCAPTCHA v3 + honeypot 필드.

---

## 10. 접근성 규칙

- 모든 인터랙티브 요소는 키보드로 접근 가능해야 합니다.
- 컬러 대비 WCAG AA 기준 충족 (본문 4.5:1, 큰 글자 3:1).
- 폼 라벨은 명시적으로 연결 (`htmlFor` 또는 `aria-labelledby`).
- 이미지 `alt` 속성 필수. 장식용 이미지는 `alt=""`로 의도적으로 표시.
- 헤딩 계층 (h1 → h2 → h3) 건너뛰지 않기.
- 시맨틱 HTML 사용 (`<nav>`, `<main>`, `<section>`, `<article>`).

## 11. 성능 규칙

- 목표: LCP 2.5초 이내, CLS 0.1 이하, Lighthouse 90점 이상.
- 폰트는 `next/font`로 최적화.
- 큰 이미지는 `next/image`의 `priority` 또는 lazy loading 적용.
- 클라이언트 번들에 큰 라이브러리를 직접 import하지 않습니다 (동적 import 활용).
- `'use client'` 컴포넌트는 가능한 작게 유지.

## 12. 작업 방식

### 12-1. 한 번에 한 가지

작업은 작은 단위로 진행합니다.

- ❌ "Home 페이지 만들어줘"
- ✅ "Home 페이지의 Hero 섹션 컴포넌트를 만들어줘"

### 12-2. 작업 전 보고

새 작업을 시작하기 전에 다음을 먼저 보고합니다.

1. 무엇을 만들 것인가
2. 어떤 파일을 수정/생성할 것인가
3. 어떤 컴포넌트를 새로 만들 것인가
4. 추가로 결정이 필요한 사항이 있는가

사용자의 확인 후 작업을 시작합니다.

### 12-3. 작업 후 보고

작업 완료 후 다음을 보고합니다.

1. 생성/수정한 파일 목록
2. 반응형 확인 결과 (모바일 375px / 태블릿 768px / 데스크탑 1280px)
3. TypeScript 에러 0 확인
4. ESLint 경고 0 확인
5. 접근성 확인 결과
6. 다음 작업 제안

### 12-4. 임의 판단 금지

다음 상황에서는 **작업을 멈추고 사용자에게 묻습니다.**

- 카피가 `CONTENT.md`에 없을 때
- 디자인 토큰에 없는 새 컬러/사이즈가 필요할 때
- 새 라이브러리가 필요할 때
- 폴더 구조에 없는 새 경로를 추가해야 할 때
- 외부 API 키, 환경 변수가 필요할 때
- e4ds.com과의 역할 분리에서 애매한 부분이 생겼을 때

## 13. 커밋 규칙

- 커밋 메시지는 한글로 작성.
- **한 커밋 = 한 작업.** 여러 작업을 하나의 커밋에 섞지 않습니다.
- 형식: `[작업 타입] 간단한 설명`

작업 타입:
- `[feat]` 새 기능/컴포넌트 추가
- `[fix]` 버그 수정
- `[style]` 스타일·반응형 보완
- `[refactor]` 리팩터링 (기능 변화 없음)
- `[docs]` 문서 업데이트
- `[chore]` 빌드·의존성·설정 변경
- `[content]` 카피·콘텐츠 변경

예시:
- `[feat] Home Hero 섹션 컴포넌트 추가`
- `[fix] Contact 폼 이메일 검증 오류 수정`
- `[style] Header 모바일 반응형 보완`
- `[content] Services 페이지 카피 CONTENT.md 기준으로 갱신`

## 17. 보충: 자주 묻는 결정 사항

다음은 향후 작업 중 자주 마주칠 결정 사항에 대한 기본 방향입니다.

- **모바일 우선** 설계. 데스크탑 → 모바일이 아니라 모바일 → 데스크탑.
- **다크모드는 1차 오픈 범위에 포함하지 않음.** 라이트 모드만 완성도 있게.
- **로그인 기능 없음.** 회사 사이트는 공개 페이지만으로 구성.
- **블로그·인사이트 페이지는 2차 오픈에서 검토.** 1차는 7개 핵심 페이지만.
- **검색 기능 없음.** 정보가 적어 메뉴 탐색으로 충분.
- **쿠키 동의 배너 필요.** GA4 사용을 위한 최소 동의 UI.
